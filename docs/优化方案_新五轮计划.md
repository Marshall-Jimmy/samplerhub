# Jima's SamplerHub - 新五轮优化方案

> 基于项目 v1.0.0 现状分析，制定 2026 年度优化计划
> 创建日期：2026-06-04

---

## 项目现状总结

### 已完成功能
- Electron + React + TypeScript 基础架构
- SQLite + Drizzle ORM 数据库（FTS5 全文搜索）
- 文件扫描与增量同步（chokidar 实时监控）
- 音频元数据解析（music-metadata：BPM/Key/时长/采样率）
- 智能分类系统（19 个预设分类 + 关键词/正则/文件夹规则）
- 采样列表展示与播放控制（Howler.js）
- DAW 拖拽导出（原生 startDrag）
- 六主题系统（黑曜石/午夜蓝/玫瑰粉/墨绿/水墨/浅色）
- 收藏夹 / 最近使用 / 标签管理
- 全局快捷键 / 搜索筛选面板
- 设置面板 / 帮助弹窗
- 启动画面（Splash Screen）
- electron-builder 打包（NSIS + Portable）

### 已知问题与瓶颈
1. **大库性能**：8000+ 文件时加载慢，列表无虚拟滚动
2. **波形不真实**：非 WAV 文件使用伪随机波形，WAV 波形解析未缓存
3. **列表交互单一**：仅列表视图，无网格/波形视图切换
4. **分类不可自定义**：分类规则硬编码在数据库初始化中，用户无法编辑
5. **无批量操作**：不支持多选、批量删除、批量移动分类
6. **无播放列表**：缺少用户自定义播放列表功能
7. **搜索体验**：无搜索建议、无搜索历史、无高亮匹配
8. **拖拽体验**：拖拽图标为应用图标而非音频波形缩略图
9. **元数据不完整**：BPM/Key 检出率低，缺少自动 BPM 检测
10. **无自动更新**：electron-updater 已引入但未配置发布源

## 第一轮：性能与稳定性优化

**目标**：解决大库卡顿问题，提升核心体验流畅度

### 1.1 虚拟滚动列表
**问题**：8000+ 采样全部渲染，DOM 节点过多导致卡顿
**方案**：
- 引入 `react-window` 的 `VariableSizeList` 替换当前全量渲染
- 使用 `react-virtualized-auto-sizer` 自适应容器高度
- 每行高度固定 48px，预估行数计算虚拟窗口
- 保留 `AnimatePresence` 仅用于播放状态切换动画

```typescript
// LibraryPage.tsx 核心改造
import { FixedSizeList as List } from 'react-window';
import AutoSizer from 'react-virtualized-auto-sizer';

const Row = ({ index, style }) => {
  const sample = samples[index];
  return (
    <div style={style}>
      <SampleCard key={sample.id} {...sampleProps} />
    </div>
  );
};

<AutoSizer>
  {({ height, width }) => (
    <List height={height} width={width} itemCount={samples.length} itemSize={48}>
      {Row}
    </List>
  )}
</AutoSizer>
```

### 1.2 数据库查询优化
**问题**：GET_SAMPLES 一次返回所有数据，无分页
**方案**：
- 实现 `samples:getPaginated` IPC 通道，支持游标分页
- 默认每页 200 条，滚动到底部自动加载下一页
- 添加 SQLite 索引：`CREATE INDEX IF NOT EXISTS idx_samples_category ON samples(category_id)`
- 添加索引：`CREATE INDEX IF NOT EXISTS idx_samples_favorite ON samples(is_favorite)`
- 添加索引：`CREATE INDEX IF NOT EXISTS idx_samples_play_count ON samples(play_count DESC)`

### 1.3 启动优化
**问题**：启动时全量加载采样列表阻塞 UI
**方案**：
- 主进程启动时先加载分类和文件夹信息，采样列表延迟加载
- 渲染进程使用 `React.lazy` + `Suspense` 懒加载非首屏组件
- 数据库初始化改为异步，不阻塞窗口创建
- 添加启动性能埋点（数据库初始化耗时、首屏渲染耗时）

### 1.4 文件扫描并行化
**问题**：当前逐文件串行解析元数据，扫描 8000 文件耗时过长
**方案**：
- 使用 Worker 线程池并行解析音频元数据
- 批量插入数据库（`INSERT ... VALUES (...), (...), (...)` 替代逐条 INSERT）
- 扫描阶段先快速收集文件列表（仅 stat），再分批解析元数据
- 添加扫描取消功能（IPC `scan:stop`）

### 1.5 内存泄漏修复
**问题**：Howler 实例未正确释放，波形数据 base64 传输占用大量内存
**方案**：
- `playerStore.play()` 确保每次 `howl.unload()` 释放旧实例
- `audio:getBuffer` 限制文件大小（>50MB 不传输 base64）
- 组件卸载时清理 `requestAnimationFrame` 和事件监听
- 添加 `WeakRef` 缓存策略，避免波形数据长期驻留内存

## 第二轮：音频体验增强

**目标**：让采样预览体验达到专业级水准

### 2.1 真实波形生成与缓存
**问题**：非 WAV 文件使用伪随机波形，WAV 波形每次重新计算
**方案**：
- 主进程使用 `ffmpeg` 或 `music-metadata` 提取所有格式的真实波形
- 波形数据存入数据库 `waveform_data` 字段（压缩为 Float32Array）
- 扫描时异步生成波形，不阻塞元数据解析
- 渲染进程从数据库读取波形，不再通过 IPC 传输 base64

```typescript
// 波形生成与存储
export async function generateWaveform(filePath: string): Promise<Float32Array> {
  const SAMPLES = 200;
  // 使用 ffmpeg 解码为 PCM
  const pcmData = await decodeToPCM(filePath);
  // 降采样为 200 个点
  const waveform = downsample(pcmData, SAMPLES);
  // 归一化
  normalize(waveform);
  return waveform;
}
```

### 2.2 播放器增强
**问题**：播放控制功能基础，缺少专业功能
**方案**：
- **A-B 循环**：选择片段循环播放，用于精细试听
- **播放速度**：0.5x / 0.75x / 1x / 1.25x / 1.5x / 2x 切换
- **音调保持**：变速不变调（Howler 不支持，需 Web Audio API）
- **波形拖拽定位**：点击波形跳转播放位置（已实现，优化精度）
- **波形缩放**：鼠标滚轮缩放波形显示范围

### 2.3 音频可视化增强
**问题**：PlayerBar 波形显示简单，缺少动态反馈
**方案**：
- 播放时波形实时跟随播放位置高亮
- 添加频谱分析器（AnalyserNode）显示实时频谱
- 波形 hover 时显示时间刻度
- 使用 `wavesurfer.js` 替换自定义 Canvas 波形（已引入但未使用）

### 2.4 自动 BPM 检测
**问题**：大量采样缺少 BPM 元数据，依赖文件标签
**方案**：
- 集成 `aubio` 或 `essentia.js` 进行 BPM 检测
- 扫描时对缺少 BPM 的文件自动检测
- 设置面板添加「自动检测 BPM」开关
- 检测结果存入数据库，不覆盖已有标签 BPM

### 2.5 音频格式兼容性
**问题**：部分格式播放失败或元数据解析不完整
**方案**：
- 添加格式检测与降级播放策略
- 不支持的格式显示提示而非静默失败
- 添加「损坏文件」标记，避免反复解析失败
- 支持预览 MIDI 文件（使用 Tone.js 合成）

## 第三轮：交互与 UI 升级

**目标**：打造专业级音乐制作工具的交互体验

### 3.1 多视图模式
**问题**：仅有列表视图，无法满足不同浏览需求
**方案**：
- **列表视图**（当前）：紧凑排列，适合快速浏览
- **网格视图**：卡片式布局，显示大波形 + 完整元数据
- **波形视图**：以波形为主体，类似 SoundCloud 风格
- 视图切换按钮放在搜索栏旁
- 视图偏好持久化到 settingsStore

```typescript
// libraryStore 添加
viewMode: 'list' | 'grid' | 'waveform';
```

### 3.2 批量操作
**问题**：无法多选采样进行批量操作
**方案**：
- 添加多选模式（Ctrl+Click / Shift+Click / Ctrl+A）
- 多选后显示浮动操作栏：批量删除、批量移动分类、批量添加标签、批量导出
- 批量删除前确认弹窗
- 支持拖拽多选文件到 DAW

### 3.3 右键上下文菜单
**问题**：所有操作依赖按钮点击，缺少快捷操作入口
**方案**：
- 自定义右键菜单（Electron `Menu` 或 React 组件）
- 菜单项：播放、收藏、编辑标签、移动分类、在资源管理器中打开、复制路径、删除
- 分类子菜单：显示所有分类供快速选择
- 标签子菜单：显示最近使用的标签

### 3.4 拖拽体验优化
**问题**：拖拽图标为应用图标，不够直观
**方案**：
- 生成采样波形缩略图作为拖拽图标
- 拖拽时显示文件名 + 时长信息
- 支持拖拽到系统文件管理器（复制文件）
- 添加拖拽预览动画

### 3.5 搜索体验增强
**问题**：搜索功能基础，缺少智能辅助
**方案**：
- **搜索建议**：输入时显示历史搜索和热门标签
- **搜索高亮**：匹配结果中高亮关键词
- **搜索历史**：记录最近 20 条搜索，支持快速重搜
- **模糊搜索**：支持拼写容错（Levenshtein 距离）
- **保存搜索条件**：常用筛选组合保存为预设

### 3.6 排序功能
**问题**：当前仅按播放次数 + 文件名排序
**方案**：
- 列表头支持点击排序：名称、时长、BPM、Key、添加时间、播放次数、文件大小
- 排序方向切换（升序/降序）
- 排序偏好持久化

## 第四轮：功能完善与智能化

**目标**：补齐核心功能短板，提升智能化水平

### 4.1 播放列表系统
**问题**：缺少用户自定义播放列表
**方案**：
- 新增 `playlists` 和 `playlist_items` 数据库表
- 侧边栏添加「播放列表」区域
- 支持创建/删除/重命名播放列表
- 拖拽采样到播放列表添加
- 播放列表内排序和移除
- 导出播放列表为 M3U/PLS 格式

```sql
CREATE TABLE playlists (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT NOT NULL,
  description TEXT,
  cover_color TEXT DEFAULT '#6366F1',
  created_at INTEGER NOT NULL DEFAULT (unixepoch()),
  updated_at INTEGER NOT NULL DEFAULT (unixepoch())
);

CREATE TABLE playlist_items (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  playlist_id INTEGER NOT NULL REFERENCES playlists(id) ON DELETE CASCADE,
  sample_id INTEGER NOT NULL REFERENCES samples(id) ON DELETE CASCADE,
  sort_order INTEGER NOT NULL DEFAULT 0,
  added_at INTEGER NOT NULL DEFAULT (unixepoch()),
  UNIQUE(playlist_id, sample_id)
);
```

### 4.2 分类规则编辑器
**问题**：分类规则硬编码，用户无法自定义
**方案**：
- 设置面板添加「分类规则」标签页
- 可视化规则编辑器：添加/编辑/删除/排序规则
- 规则测试面板：输入文件名预览匹配结果
- 支持导入/导出规则配置（JSON）
- 规则优先级拖拽排序

### 4.3 智能推荐
**问题**：缺少基于使用习惯的推荐
**方案**：
- **相似采样推荐**：基于 BPM + Key + 分类匹配相似采样
- **经常一起使用**：基于播放序列统计关联采样
- **最近趋势**：展示近期最常播放的采样类型
- 推荐结果在侧边栏或首页展示

### 4.4 采样详情面板
**问题**：采样信息分散，缺少集中展示
**方案**：
- 右侧可展开的详情面板
- 显示完整元数据：文件路径、大小、时长、采样率、位深度、声道、BPM、Key
- 大波形预览（可交互）
- 标签编辑
- 分类修改
- 播放历史
- 关联采样推荐

### 4.5 文件管理增强
**问题**：无法在应用内管理文件
**方案**：
- 「在资源管理器中打开」功能
- 文件重命名（同步更新数据库）
- 文件移动（拖拽到新文件夹）
- 重复文件检测（基于 fileHash）
- 损坏文件标记与清理

### 4.6 导入/导出
**问题**：数据无法迁移
**方案**：
- 导出数据库为 JSON（采样信息 + 标签 + 播放列表 + 分类规则）
- 导入 JSON 恢复数据（合并策略：跳过/覆盖/重命名）
- 导出采样文件列表（CSV/TSV）
- 批量复制采样文件到指定目录

## 第五轮：打磨与发布

**目标**：产品级打磨，准备正式发布

### 5.1 自动更新
**问题**：electron-updater 已引入但未配置
**方案**：
- 配置 GitHub Releases 作为更新源
- 实现增量更新（delta update）
- 更新检查策略：启动时静默检查，有更新时提示
- 更新进度显示（已实现 update 组件）
- 支持跳过版本、延迟更新

### 5.2 错误处理与日志
**问题**：错误处理不完善，难以排查问题
**方案**：
- 全局错误边界（React ErrorBoundary）
- 主进程日志系统（electron-log）
- 日志文件轮转（最大 10MB，保留 5 个文件）
- 崩溃报告收集（可选）
- 用户反馈入口（GitHub Issues 链接）

### 5.3 无障碍与国际化
**问题**：无 ARIA 标签，仅支持中文
**方案**：
- 添加 ARIA 标签和键盘导航支持
- 使用 `react-intl` 或 `i18next` 实现国际化
- 首批支持语言：中文、英文
- 日期/数字格式本地化

### 5.4 测试覆盖
**问题**：测试覆盖不足
**方案**：
- 单元测试：分类算法、搜索逻辑、波形计算（Vitest）
- 集成测试：IPC 通信、数据库操作
- E2E 测试：核心用户流程（Playwright，已有配置）
- 性能基准测试：大库加载、搜索响应时间
- 目标覆盖率：核心逻辑 > 80%

### 5.5 性能监控
**问题**：无性能数据，优化缺乏依据
**方案**：
- 添加性能埋点：启动时间、列表渲染时间、搜索响应时间
- 内存使用监控（`process.memoryUsage()`）
- 数据库查询耗时统计
- 渲染帧率监控（开发模式）
- 性能数据写入本地日志，可选上报

### 5.6 打包与分发优化
**问题**：安装包体积大，分发渠道单一
**方案**：
- 启用 ASAR 压缩优化
- 排除开发依赖（`devDependencies` 不打包）
- 添加应用签名（Windows Code Signing）
- 配置多渠道分发：GitHub Releases、官网下载
- 添加便携版配置（已支持 Portable）
- macOS 签名与公证（Apple Developer ID）

### 5.7 文档与社区
**方案**：
- 完善用户文档（使用指南、FAQ）
- 开发者文档（架构说明、贡献指南）
- 录制功能演示视频
- GitHub README 双语完善

## 实施优先级

| 优先级 | 轮次 | 核心价值 | 预估复杂度 |
|--------|------|----------|-----------|
| P0 | 第一轮 | 解决大库卡顿，保障基础可用 | 中 |
| P0 | 第二轮 | 音频体验是核心卖点 | 高 |
| P1 | 第三轮 | 交互体验决定用户留存 | 中 |
| P1 | 第四轮 | 功能完善提升产品完整度 | 高 |
| P2 | 第五轮 | 打磨细节，准备发布 | 中 |

## 技术债务清单

| 项目 | 严重程度 | 说明 |
|------|---------|------|
| SampleCard 内联样式过多 | 低 | 应抽取为 styled-component 或 CSS Module |
| playerStore 直接引用 Howler | 中 | 应抽象播放器接口，便于替换实现 |
| IPC 类型安全不完整 | 中 | 部分 IPC 通道缺少类型约束 |
| 数据库迁移策略缺失 | 高 | schema 变更无版本管理 |
| 无错误边界 | 高 | 组件异常导致白屏 |
| 主题 CSS 变量与 Ant Design 重复 | 低 | 应统一为一套主题系统 |
